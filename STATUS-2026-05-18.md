## TL;DR — where you'll pick up

After 9 Docker builds spent fighting pip's transitive torch upgrades and failing every time, we **pivoted to the SSH-into-Pod approach**: no Docker building, use RunPod's stock pytorch template, run a setup script that installs the missing Python packages without disturbing torch.

When you're ready to test, the steps are:

1. **Deploy a fresh Pod** with template `runpod/pytorch:2.4.0-py3.11-cuda12.4.1-devel-ubuntu22.04` (the same template you've been using for temp Pods to verify weights)
2. **Open the Pod's Web Terminal** and run **one curl command** (setup script)
3. **Start the server** with one nohup command
4. **Update `.env`** with the new Pod URL
5. **Restart local uvicorn**
6. **Submit a character-swap test**

Total wall-clock time: ~10 minutes (mostly waiting for the Pod to boot and for setup_wan_pod.sh to pip-install ~5 packages).

## The setup script — what it does

`runpod-wan-animate/setup_wan_pod.sh` lives on GitHub now. From inside a fresh Pod:

```bash
curl -fsSL https://raw.githubusercontent.com/kpashoks/replicate-clone/main/runpod-wan-animate/setup_wan_pod.sh | bash
```

The script:

1. **Sanity-checks** — verifies torch + CUDA + GPU + all 4 weight files
2. **Captures pre-installed torch version** — so it can verify it wasn't upgraded at the end
3. **Clones Wan-Video/Wan2.2** to `/opt/wan22`
4. **Installs FastAPI deps** (with normal dep resolution, no torch involved)
5. **Installs Wan model deps** (`diffusers==0.31.0`, `transformers==4.51.3`, `accelerate==1.1.1`, `tokenizers==0.20.3`) — pinned to versions whose own dependencies don't pull a newer torch. Uses `--no-deps` as belt-and-suspenders.
6. **Installs Meta SAM2** pinned to commit `2b90b9f5` (Oct 2024, before SAM2 bumped its torch requirement to `>=2.5.1`)
7. **Verifies torch was NOT upgraded** — exits non-zero with a clear error if it was
8. **Smoke-tests imports** — `import torch`, `import diffusers`, `import sam2`, `import wan.configs`, etc.
9. **Downloads `server.py`** from this repo to `/opt/wan-animate/server.py`
10. **Writes `/opt/wan-animate/start.sh`** with the right env vars
11. **Writes `/opt/wan-animate/update_server.sh`** for quick iteration on `server.py` later

## Why this approach should finally work

| Previous problem | What's different now |
|---|---|
| pip silently upgraded torch from 2.4.1+cu121 to 2.12.0+cu130 through transitive resolution | Pre-installed torch is preserved; we use pinned dep versions + `--no-deps` |
| Custom Docker image with stale Python 3.11 (Ubuntu 22.04's pre-release) | RunPod's stock template has Python 3.11.X (current stable) |
| `:latest` tag caching across rebuilds | No image being built — install fresh on each Pod |
| Slow Docker rebuild loop (~12 min per iteration) | In-Pod iteration via web terminal — seconds |
| Web terminal didn't work on python:3.11-slim | Works on this template (has SSH daemon) |
| SAM2 latest requires torch>=2.5.1, would force upgrade | Pinned to Oct 2024 commit before that bump |

If something STILL fails after running the setup script, you have a working Web Terminal to debug live. Edit `/opt/wan-animate/server.py`, restart uvicorn, test. No image rebuilds.

## Step-by-step when you're back

### Step 1 — Deploy a fresh Pod

RunPod console → Pods → Deploy:

| Field | Value |
|---|---|
| **Template** | `runpod/pytorch:2.4.0-py3.11-cuda12.4.1-devel-ubuntu22.04` |
| **GPU** | RTX 6000 Ada (the one you had before is fine) |
| **Region** | US-KS-2 (same as your volume) |
| **Network Volume** | Attach `replicate-local-models` at **`/workspace`** |
| **Container disk** | 30 GB |
| **Expose HTTP port** | 8000 |
| Web terminal | enabled |

Click Deploy. Wait ~30 sec for it to boot.

### Step 2 — Open the Pod's Web Terminal

From the Pod's page, click the Web Terminal toggle, then "Open web terminal".

### Step 3 — Run the setup script

In the terminal:

```bash
curl -fsSL https://raw.githubusercontent.com/kpashoks/replicate-clone/main/runpod-wan-animate/setup_wan_pod.sh | bash
```

Watch for the green `[setup]` log lines. The script should run for ~2-3 minutes (pip downloads dominate). It ends with `Setup complete.` and a block of next-step instructions.

**If it exits non-zero**, paste the error here. The script is loud about what failed.

### Step 4 — Start the server

```bash
nohup /opt/wan-animate/start.sh > /var/log/wan-animate.log 2>&1 &
tail -f /var/log/wan-animate.log
```

Expected log output:
```
[start.sh] Python 3.11.X | torch 2.4.0+cu124 | CUDA 12.4 | CUDA OK: True
INFO:     Started server process [N]
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8000
```

Ctrl+C the `tail` command — server keeps running because of `nohup ... &`.

### Step 5 — Verify the server in your browser

Get the Pod's port-8000 public URL from the Connect tab. Hit:

```
https://<pod-id>-8000.proxy.runpod.net/debug/info
```

Expected response (the values you want to see are highlighted):

```json
{
  "python": "3.11.X",
  "torch": "2.4.0+cu124",            ← MUST be 2.4.0, NOT 2.12.0
  "torch_cuda_version": "12.4",
  "cuda_available": true,             ← MUST be true
  "cuda_device_count": 1,
  "cuda_device_name": "NVIDIA RTX 6000 Ada Generation",
  "wan_repo_exists": true,
  "wan_repo_path": "/opt/wan22",
  "wan_repo_commit": "<some hash>",
  "sam2_path": "..."                 ← MUST be non-null
}
```

**If torch shows `2.4.0+cu124` and `cuda_available: true`**, you're past every CUDA hurdle. Proceed to Step 6.

**If torch shows anything else** (especially `2.12.0+cu130`), the script's torch-preservation check should have caught it — paste the script's output.

### Step 6 — Wire into local app

In `C:\Users\kpash\projects\replicate-local\.env`, set:

```
WAN_ANIMATE_ENDPOINT=https://<pod-id>-8000.proxy.runpod.net
```

(Replace `<pod-id>` with the actual Pod ID.)

### Step 7 — Restart local uvicorn

Full restart, not `--reload`:

```
Ctrl+C  (in the uvicorn terminal)
uvicorn main:app --reload    (from backend/ directory)
```

### Step 8 — Submit a character-swap test

In the frontend at `http://localhost:3000`:

1. Click **Character Swap (Video)**
2. Upload your source video + character reference image
3. Resolution: **832x480** (fastest, biggest chance of first-try success)
4. `replace_flag`: **true** (Mix mode)
5. Click **Run**

While that runs, **keep the Pod's Logs tab open** (or `tail -f /var/log/wan-animate.log` in the web terminal). You'll see the actual Wan model load progression in real time:


## What could still go wrong

If the setup script succeeds but the FIRST character-swap submit produces an error, it'll be Wan-Video-specific (not CUDA / Docker / Python-version):

| Likely error | Fix |
|---|---|
| `KeyError: 'animate-14B'` in WAN_CONFIGS lookup | `server.py` line 188: change to whatever key Wan actually uses. SSH in, `nano /opt/wan-animate/server.py`, edit, run `/opt/wan-animate/update_server.sh` |
| `ImportError: cannot import name 'WanAnimate' from 'wan'` | Class path different. Same fix path. |
| `ProcessPipeline` not found in preprocessing module | The introspection fallback in `_load_model()` will pick a class — verify it's the right one |
| `FileNotFoundError` on a weight path | Wan expects weights at a specific path — adjust `WAN_CKPT_DIR` env var or symlink files |
| CUDA OOM | Drop to 832x480 resolution if you used something higher; or pick bigger GPU |

For ANY of these, the iteration loop is now:
1. Read the error from `/var/log/wan-animate.log` in the Pod's web terminal
2. Edit `/opt/wan-animate/server.py` directly (`nano` or `vim`)
3. Kill the running uvicorn: `pkill -f 'uvicorn.*server:app'`
4. Restart: `nohup /opt/wan-animate/start.sh > /var/log/wan-animate.log 2>&1 &`
5. Retest

OR push the fix to GitHub and run `/opt/wan-animate/update_server.sh` which does pull + restart in one command.
